# Trading System Spec Adoption Plan

本项目当前是“群消息识别 + 新手短线风控工作流”，不是完整自动交易平台。`AGENTS.md` 和 `docs/trading-system-spec.md` 已作为长期规格引入，但落地顺序必须服从当前用户画像：资金约 34,000 元、A 股新手、先做识别、核验、风控、复盘，不自动下单。

## 采纳原则

1. 不把完整交易系统一次性重构进当前项目。
2. 先加强已有风控链路，再引入指标、市场状态和回测。
3. 任何买入结论必须保留硬性否决：缺价格、超过目标、涨停、止损无效、盈亏比不足、一手风险超限。
4. 群消息仍只作为线索，不能成为 A 级证据。
5. 回测和策略层上线前，先累计至少 20 条真实群消息复盘样本。

## 当前系统与规格映射

| 规格领域 | 当前状态 | 下一步 |
| --- | --- | --- |
| 数据层 | 已有东方财富/腾讯行情、消息时点分时价、手动数据源接口 | 增加数据质量对象、缓存目录、数据源健康检查 |
| 指标层 | 已有 5/20 日均价、涨跌幅、振幅、量比、ATR14、RSI14、MACD 等轻量体检 | 建立最小指标注册器，统一 MA、ATR、RSI、MACD、量比 |
| 信号层 | 已有群消息解析、红线、证据核验、机会评级 | 引入标准化 `Signal`，区分群消息线索和技术信号 |
| 决策层 | 已有硬性否决、机会评级、系统建议止盈止损 | 增加决策日志和分项权重，不做黑箱评分 |
| 风控层 | 已有一手成本、训练仓、单笔亏损、建议止盈止损、ATR 止损候选、组合仓位上限 | 增加移动止损和行业集中度约束 |
| 市场状态 | 尚未实现 | 先实现基于指数 MA60、涨跌家数、成交量的简化五状态 |
| 多时间框架 | 尚未实现 | 先用周线/日线方向做信号降级，不直接升级买入 |
| 策略层 | 尚未作为独立策略注册器实现 | 等样本和指标稳定后实现趋势/均值/突破策略 |
| 回测层 | 已有复盘样本评分，未做历史策略回测 | 先做群源样本回测，再接 vectorbt/backtrader |
| 输出层 | Markdown 报告、CLI、records | 增加 JSON 输出和日报模板 |
| 配置层 | 新增 `config/config.yaml` 模板，代码尚未读取 | 后续增加配置加载和范围校验 |

## P0：不改变行为的工程吸收

- 保留现有 `stock_recognition_system` 包结构，不迁移到 `src/`。
- 将 `AGENTS.md`、`docs/trading-system-spec.md`、`config/config.yaml` 纳入仓库。
- 新增本采纳计划，明确当前系统不做自动交易。
- 继续用 `unittest`，后续若切 pytest 需单独迁移。

## P1：短线风控增强

1. 配置加载：读取 `config/config.yaml` 中的账户、训练仓、止损、止盈、数据源参数。
2. 数据质量：为行情 provider 输出数据源、时间、字段完整性、失败降级原因。
3. 行业集中度：同题材/同行业多票同时持仓时只降级，不自动升级买入。
4. 市场状态简版：只用于降级仓位，不用于自动升级买入。
5. JSON 报告：让后续 Web 页面或表格复盘能读取结构化结果。

## P2：信号和策略框架

1. `Signal` dataclass：标准化群消息信号、技术信号、证据信号。
2. 指标注册器：先实现 MA、ATR、RSI、量比。
3. 策略注册器：先实现观察型策略，不输出真实买入。
4. 多时间框架：周线和日线冲突时降级机会评级。

## P3：回测与产品化

1. 用 `records/outcomes.jsonl` 做群源历史样本复盘。
2. 扩展到交易成本、滑点、次日开盘成交假设。
3. 样本达到 50 条后再讨论参数优化。
4. 样本达到 100 条后再考虑策略回测框架。

## 当前不做

- 不自动下单。
- 不接真实券商交易接口。
- 不用大模型或群消息直接给买入指令。
- 不在样本不足时做“胜率证明”。
- 不因为某个股票后续上涨就回头放宽所有规则。
