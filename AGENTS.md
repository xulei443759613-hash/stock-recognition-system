# 股票买卖决策工作流 — 项目规范与 Codex 协作指南

> 本文件是 Codex 在本项目中的最高指导文档。Codex 在每个阶段开始前必须阅读对应章节，理解目标、思维原则和具体指令后再动手写代码。

---

## 1. 项目概述

### 1.1 目标

构建一个**股票买卖决策工作流系统**，覆盖从市场数据获取、技术/基本面信号分析、买卖决策生成、风险控制到回测验证的完整链路。

### 1.2 用户画像

- 角色：程序员 + 股票交易新手
- 技术能力：能写代码、能读代码、能跑测试，但缺乏金融领域深度知识
- 核心诉求：系统必须能"行之有效"——不只是代码能跑，而是决策逻辑本身在回测和实盘中经得起检验

### 1.3 核心原则

| 原则 | 含义 |
|------|------|
| 数据准确性 | 错误的数据会导致错误的决策，数据层必须做严格校验 |
| 信号可解释性 | 每个买卖信号必须能说清"为什么"，黑箱信号不可接受 |
| 风控优先 | 任何决策在发出前必须过风控检查，宁可错过不要亏错 |
| 回测可信 | 回测结果必须避免未来函数、过拟合等陷阱 |

## 2. Codex 协作总则

在进入具体阶段之前，Codex 必须遵守以下四条协作原则。这些原则贯穿所有阶段。

### 2.1 先方案后代码

Codex 在动手写任何代码之前，必须先用文字描述将要做什么、为什么这样做、有哪些备选方案。等用户确认方案后，再进入编码。如果用户直接要求"写代码"，Codex 应主动先简要说明方案（3-5 句话），然后开始编码。

### 2.2 小步快跑

每次只完成一个可独立验证的小任务（一个函数、一个测试、一段配置）。不要一次性生成整个模块。完成后立即让用户验证，验证通过再进行下一步。这样做的好处是：错误可定位、可控，不会积累技术债。

### 2.3 上下文先行

Codex 在回答任何问题前，必须先读取相关的项目文件，理解现有代码结构、命名规范和已有实现。不要凭空假设项目结构。如果缺少必要的上下文，Codex 应主动询问或使用文件搜索工具先了解现状。

### 2.4 交叉验证

金融领域 API 和数据容易出错（数据源变更、字段含义变化、时区问题）。Codex 在使用任何金融数据 API 或计算逻辑时，必须：
- 查阅官方文档确认字段含义
- 用已知数据（如某只股票某天的收盘价）做交叉验证
- 对关键计算（如收益率、夏普比率）给出公式来源

## 3. 项目阶段与 Codex 指令

### 阶段一：需求定义

#### 目标
明确系统要解决什么问题，输入输出是什么，边界和约束在哪里。

#### 思维原则
- **用户视角**：从"一个普通股民每天需要什么信息来做决策"出发，而不是从"技术上能做什么"出发
- **边界思维**：明确系统**不做什么**——不提供投资建议、不自动下单、不预测未来价格

#### Codex 指令模板

```
我们要构建一个股票买卖决策工作流系统。请帮我梳理以下问题：
1. 系统的核心功能边界（做什么、不做什么）
2. 目标用户的使用场景（每日盘中？盘后分析？）
3. 输入数据来源和格式
4. 输出形式（报告？信号？可视化？）
5. 关键约束（实时性要求、数据准确性要求、可解释性要求）

请先不要写代码，只输出需求文档。
```

#### 交付物
- `docs/requirements.md`：需求文档，包含功能边界、使用场景、输入输出定义、约束清单

#### 验收标准
- 每条需求都可验证（不是"系统要好用"，而是"系统能输出包含5个技术指标的买卖信号报告"）
- 边界清晰（明确列出"不做什么"）

---

### 阶段二：架构设计

#### 目标
确定技术选型、模块划分、数据流、接口定义。

#### 思维原则
- **权衡思维**：没有完美方案，只有适合场景的方案。Python 适合快速原型和数据分析，但如果需要低延迟交易则不合适
- **可扩展性**：模块之间通过清晰接口通信，新指标、新数据源可以即插即用，但不过度设计

#### Codex 指令模板

```
基于需求文档（docs/requirements.md），请设计系统架构：
1. 技术栈选择（语言、框架、数据存储）及理由
2. 模块划分和数据流图（用文字描述或 mermaid 图）
3. 核心接口定义（每个模块的输入输出）
4. 目录结构设计

约束：
- 用户是 Python 开发者
- 优先使用成熟的开源库（pandas, akshare/tushare, backtrader/vectorbt 等）
- 模块间松耦合，方便单独测试和替换
```

#### 交付物
- `docs/architecture.md`：架构文档
- 项目目录结构

#### 验收标准
- 每个模块职责单一（数据获取不包含信号计算，信号计算不包含风控逻辑）
- 接口定义清晰，输入输出类型明确
- 数据流可以从"数据源"一路追踪到"最终输出"

---

### 阶段三：任务拆解

#### 目标
把架构设计中的模块拆成可独立完成、可独立验证的小任务。

#### 思维原则
- **分治思维**：每个任务都应小到"一次能做完且能验证"。如果一个任务需要超过 2 小时，继续拆

#### Codex 指令模板

```
基于架构文档（docs/architecture.md），请把项目拆解为有序的任务列表：
1. 每个任务包含：名称、目标、输入、输出、依赖任务、验收标准
2. 标注任务间的依赖关系（哪些必须先完成）
3. 标注优先级（P0 = 核心链路必须先做，P1 = 增强功能，P2 = 可选优化）

输出格式为表格，方便我逐个推进。
```

#### 交付物
- `docs/tasks.md`：任务清单表

#### 验收标准
- 每个任务的验收标准是可测试的（能写出一个具体的测试用例来验证）
- 依赖关系无环（不存在 A 依赖 B、B 又依赖 A）

---

### 阶段四：编码实现

#### 目标
按任务清单逐个实现，每个任务完成后立即验证。

#### 思维原则
- **抽象思维**：识别重复逻辑，抽取通用函数。比如多个技术指标都需要"滚动窗口计算"，应抽取为公共工具
- **防御性编程**：外部数据不可信。API 可能返回空值、NaN、异常格式。每个数据入口必须做校验和清洗

#### Codex 指令模板（每个任务通用）

```
请实现任务：[任务名称]
参考文档：docs/tasks.md 中该任务的描述

要求：
1. 先用 2-3 句话说明你的实现思路
2. 遵循项目现有的命名规范和目录结构
3. 函数必须有类型注解和 docstring
4. 关键逻辑加注释说明"为什么这样做"
5. 同时编写对应的单元测试
6. 处理边界情况：空数据、NaN、异常输入
```

#### 编码规范

| 规范 | 要求 |
|------|------|
| 命名 | 函数和变量用 snake_case，类用 CamelCase，常量用 UPPER_SNAKE |
| 类型注解 | 所有函数参数和返回值必须有类型注解 |
| Docstring | 每个公共函数必须有 docstring，说明参数、返回值、异常 |
| 错误处理 | 不吞异常，不静默失败。异常要么处理并记录，要么向上抛出 |
| 日志 | 关键操作（数据获取、信号生成、决策输出）必须有日志，用 logging 模块 |
| 配置 | API 密钥、股票池、参数等放配置文件，不硬编码在代码中 |

### 阶段五：测试验证

#### 目标
确保每个模块和整个系统的行为符合预期。

#### 思维原则
- **证伪思维**：测试的目的是找 bug，不是证明代码没问题。重点测试边界情况和异常路径

#### Codex 指令模板

```
请为 [模块/函数名] 编写测试用例，覆盖以下场景：
1. 正常路径：典型输入，验证输出正确
2. 边界情况：空数据、单个数据点、最小/最大值
3. 异常输入：NaN、None、错误类型、越界索引
4. 金融逻辑验证：用已知数据验证计算结果（如已知收盘价序列验证 MA 计算正确）

如果发现实现中有 bug，请先修复再继续测试。
```

#### 测试要求

| 层级 | 覆盖范围 | 工具 |
|------|---------|------|
| 单元测试 | 每个函数的输入输出和边界 | pytest |
| 集成测试 | 模块间数据流（数据获取→信号计算→决策生成） | pytest |
| 回测验证 | 用历史数据验证策略逻辑，检查是否有未来函数 | vectorbt / backtrader |
| 数据验证 | 用已知数据交叉验证 API 返回值的正确性 | 手动 + 自动 |

### 阶段六：调试排错

#### 目标
复现问题，定位根因，修复，验证。

#### 思维原则
- **溯源思维**：沿调用链从报错点往上找，找到最早出错的地方
- **二分法**：通过注释代码或加日志，缩小问题范围

#### Codex 指令模板

```
遇到以下问题：[粘贴错误信息/异常行为描述]

已尝试的方法：[列出你已经排查过的方向]

请按以下顺序帮我排查：
1. 分析错误信息的直接含义
2. 追踪调用链，找出最可能出错的环节
3. 给出排查步骤（先做什么、后做什么）
4. 如果需要更多信息，明确告诉我需要什么

注意：不要只给一个可能的答案，列出所有可能性并按概率排序。
```

### 阶段七：优化重构

#### 目标
在不改变行为的前提下，提升性能、可读性、可维护性。

#### 思维原则
- **测量优先**：先 profile 找到真正的瓶颈，再优化。不要凭感觉优化
- **长期主义**：为未来的自己写代码——命名清晰、逻辑直观、注释到位

#### Codex 指令模板

```
请审查 [模块/文件名] 的代码质量，从以下维度评估并给出改进建议：
1. 性能：是否有不必要的循环、重复计算、可向量化的操作
2. 可读性：命名是否清晰、逻辑是否直观、注释是否到位
3. 可维护性：是否便于修改和扩展
4. 健壮性：异常处理是否完善、边界情况是否覆盖

请按"收益/成本比"从高到低排序建议。只给建议，暂不修改代码。
等我确认后再逐个修改。
```

## 4. 股票交易工作流的领域约束

以下约束是金融领域的特殊要求，Codex 在实现任何相关模块时必须遵守。**详细规格参见 `docs/trading-system-spec.md`**，本文档列出约束概要，规格文档包含完整公式、参数范围、接口定义和代码示例。

### 4.1 数据层

| 约束 | 说明 |
|------|------|
| 数据源校验 | API 返回的数据必须校验：日期是否连续、价格是否合理（非负、无异常跳变）、成交量是否为正 |
| 复权处理 | 必须明确使用前复权还是后复权，并在文档和代码中标注。回测和实盘必须一致 |
| 时区 | 所有时间戳必须带时区信息，避免跨时区比较出错 |
| 缓存 | 历史数据应本地缓存，避免重复请求 API。缓存需标注数据日期，过期数据需重新获取 |
| 数据缺失 | 遇到停牌、退市等情况，数据会有缺口。系统必须能优雅处理，而非崩溃或用错误数据填充 |
| 字段规范 | 必需字段：date/open/high/low/close/volume/amount。复权字段和基本面字段按需获取 |

> 详细规格：数据字段定义、数据源对比表、校验规则细则、缓存目录设计 → `docs/trading-system-spec.md` 第 1 节

### 4.2 指标层

| 约束 | 说明 |
|------|------|
| 指标定义 | 每个技术指标必须注明计算公式和数据来源，不能只写"计算 RSI"而不说明周期和公式 |
| 参数可配置 | 所有指标参数（周期、阈值）必须可配置，不硬编码 |
| 指标注册器 | 所有指标注册到 `IndicatorRegistry`，通过名称字符串调用，支持配置驱动 |
| 接口契约 | 统一接口：输入 OHLCV DataFrame，输出 Series 或 DataFrame，前 n-1 个值为 NaN |

> 详细规格：MA/EMA/MACD/ADX/RSI/KDJ/CCI/布林带/ATR/OBV/VR/量比 共 12 个指标的完整公式、参数、信号逻辑 → `docs/trading-system-spec.md` 第 2 节

### 4.3 信号层

| 约束 | 说明 |
|------|------|
| 信号可解释 | 每个买卖信号必须附带触发原因（如"MA5 上穿 MA20，且成交量放大 1.5 倍"） |
| 无未来函数 | 信号计算只使用当前时点及之前的数据，绝不使用未来数据。这是回测可信的底线 |
| 信号标准化 | 信号必须是 `Signal` dataclass，包含股票代码、日期、类型、强度、原因、指标值 |
| 信号去重 | 同一指标同一交易日只生成一个信号（取最强） |
| 信号冷却 | 同一信号类型在 N 个交易日内不重复触发（默认 N=5） |
| 信号有效期 | 信号生成后 M 个交易日内有效（默认 M=3），过期不参与决策 |

> 详细规格：Signal 数据结构、SignalGenerator 接口、强度评级标准 → `docs/trading-system-spec.md` 第 3 节

### 4.4 决策层

| 约束 | 说明 |
|------|------|
| 决策可追溯 | 每个买卖决策必须记录：触发了哪些信号、各自权重、最终决策逻辑 |
| 多信号融合 | 不应单一信号做决策。提供三种融合方式：加权投票法、条件组合法、打分排序法 |
| 冲突处理 | 当买入信号和卖出信号同时出现时，按优先级处理：止损 > 风控否决 > 强信号 > 趋势类 > 默认持有 |
| 决策可解释 | 每条决策必须包含人类可读的 reasoning 字段：触发了哪些信号、各分项得分、风控结果 |
| 市场状态联动 | 决策必须考虑当前市场状态，不同状态使用不同策略权重。熊市自动降低仓位上限 |

> 详细规格：决策流程、融合方法公式、冲突处理优先级、Decision 输出结构 → `docs/trading-system-spec.md` 第 4 节

### 4.5 风控层

| 约束 | 说明 |
|------|------|
| 三层风控 | 个股风控（止损止盈、仓位上限、黑名单）→ 组合风控（总仓位、行业集中度、相关性）→ 系统性风控（大盘状态、回撤控制、熔断） |
| 止损方式 | 支持四种：固定百分比、ATR 动态、移动止损、均线止损。通过配置选择 |
| 仓位管理 | 支持三种：固定比例、凯利公式简化版、波动率倒数加权 |
| 风控优先级 | 风控检查必须在决策发出前执行，风控不通过则不发信号 |
| 股票池筛选 | 每日动态筛选：排除 ST、次新股、低流动性、涨跌停、退市风险股 |

> 详细规格：风控层级、止损公式、仓位计算方法、新手参数建议、股票池筛选规则 → `docs/trading-system-spec.md` 第 5 节

### 4.6 市场状态识别

| 约束 | 说明 |
|------|------|
| 五状态分类 | 强牛市 / 弱牛市 / 震荡市 / 弱熊市 / 强熊市，每种状态对应不同的策略权重和仓位建议 |
| 判断指标 | 上证指数 MA60 + 均线斜率 + 市场广度 + 涨跌家数比 + 成交量趋势 |
| 状态切换确认 | 状态切换需连续 N 天确认（默认 N=3），避免频繁切换 |
| 策略联动 | 不同市场状态下策略权重自动调整：牛市偏趋势跟踪，震荡市偏均值回归，熊市偏空仓 |

> 详细规格：状态定义、判断指标表、切换规则、策略权重调整 → `docs/trading-system-spec.md` 第 6 节

### 4.7 多时间框架分析

| 约束 | 说明 |
|------|------|
| 三级框架 | 大趋势（周线/月线，权重 50%）+ 中趋势（日线，35%）+ 小趋势（60分钟线，15%） |
| 信号对齐 | 三框架方向一致时信号升级为 STRONG，不一致时降级为 WEAK 或不发信号 |
| 可配置开关 | 多时间框架分析可通过配置开关，默认开启 |

> 详细规格：三屏交易法、多时间框架信号对齐规则 → `docs/trading-system-spec.md` 第 7 节

### 4.8 策略层

| 约束 | 说明 |
|------|------|
| 策略注册器 | 所有策略实现统一 `Strategy` 接口，注册到 `StrategyRegistry`，通过配置切换 |
| 策略适用性 | 每个策略必须声明适用的市场状态，在不适用的市场中不触发信号 |
| 内置策略 | P0 核心策略：趋势跟踪、均线交叉；P1 增强：均值回归、突破、动量；P2 可选：多因子 |

> 详细规格：策略接口、5 个内置策略的完整逻辑和参数、策略适用性矩阵 → `docs/trading-system-spec.md` 第 9 节

### 4.9 回测层

| 约束 | 说明 |
|------|------|
| 未来函数检查 | 回测框架必须自动检测未来函数，发现则报错而非静默继续 |
| 成交价模拟 | 回测使用信号日次日开盘价成交，加滑点，不使用信号日收盘价 |
| 交易成本 | 回测必须包含佣金、印花税、过户费、滑点，不包含成本的回测结果不可信 |
| 过拟合防护 | 参数优化必须使用样本内外分离，推荐使用 Walk-Forward 滚动窗口分析 |
| 绩效指标 | 必须包含：年化收益率、最大回撤、夏普比率、索提诺比率、胜率、盈亏比、卡尔玛比率。每个指标注明计算公式和合格标准 |
| 幸存者偏差 | 股票池必须包含已退市股票，否则回测结果会虚高 |

> 详细规格：回测流程、成交价模拟、未来函数检测表、样本内外分离、Walk-Forward 分析、绩效指标公式和合格标准 → `docs/trading-system-spec.md` 第 8 节

### 4.10 输出层

| 约束 | 说明 |
|------|------|
| 报告格式 | 决策报告应同时支持控制台输出和文件输出（HTML/CSV/JSON） |
| 可读性 | 报告面向非专业用户，避免堆砌术语，关键指标附简短解释 |
| 可审计 | 每份报告包含生成时间、数据范围、策略版本、参数快照 |
| 日报模板 | 包含市场概况、持仓检查、今日信号、买入候选、风控状态、决策日志六个部分 |
| 回测报告 | 包含绩效概览、交易统计、月度收益、回撤分析、参数敏感性、结论建议 |

> 详细规格：日报模板、回测报告模板 → `docs/trading-system-spec.md` 第 10 节

### 4.11 配置层

| 约束 | 说明 |
|------|------|
| 集中配置 | 所有可配置参数集中在 `config/config.yaml`，按模块组织，代码中不硬编码业务参数 |
| 配置校验 | 系统启动时校验：必填项、类型、范围（概率在 [0,1]）、一致性（策略依赖的指标已定义、权重和为 1） |
| 合理性警告 | 止损 < 1% 或 > 20%、仓位 > 50% 等极端值给出警告 |

> 详细规格：完整 config.yaml 结构、配置校验规则 → `docs/trading-system-spec.md` 第 11 节
